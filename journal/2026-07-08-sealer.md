## Feito

- Escrita via `superpowers` da spec + plano de implementação da **SEALER-01** (`point-cloud-processor`): acumulador Redis por VIN inspirado no `StrokeAccumulator` do DieMaster, `model_type → expected_frames` numa hash `settings` no Redis DB3 (fallback timeout-only se ausente), shift de câmera como posição absoluta do encoder, os 3 outputs (nuvem agregada, depth map, imagem stitchada) já na Fase 1. Plano entregue para implementação por colega (William).
- **Migração v3 do pipeline de centerlines**: os 5 scripts de pesquisa do PMO (`cap_seeded_centerlines`, `refine_centerlines_v3`, `interface_guided_centerlines`, `detect_end_faces`, `export_sealer_centerline`) foram portados fielmente para `toolkit/sealer-provisioning` (Fases A e B), validados **byte-exatos** contra os dados reais BC4B_CUV, e só então removidos do PMO — resultados/`analysis*` preservados.
- **Fase C — GUI de correção manual**: desenhada com base nos apps de referência `sdk-blender-tools/viewer` e `sdk-inspection-grouping-optimizer` (fork-and-adapt, Electron+Vite+Three.js, bridge por arquivo, não HTTP). Executada via subagent-driven TDD: Phase 0 do engine (three-pass `extract_beads`, two-pass de espelho → **227/227 byte-exato**, override `manual_points`, dependency closure) + novo repo `sdk-sealer-provisioning-gui` com viewer C1 (somente leitura) e loop de correção de caps C2 (bridge Express→CLI, snap-to-candidate), ambos validados com Playwright.
- Corrigido gap de robustez encontrado no code review: validação de schema de overrides (`_validate_overrides`) para não descartar silenciosamente correções malformadas do operador.
- Reconciliado o áudio de status do William contra o ClickUp: 8 itens do ROADMAP `[4] Software` sincronizados para done/blocked/on hold, 2 itens novos criados (pixel-to-object multi-profile, otimização point-cloud <45s), sequenciados para o William a partir de 13/07.

## Decisões

- Pipeline v3 (cap-seeded march) é o canônico; caminho antigo PCA/robot-path aposentado.
- Passo de interface-grouping mantido, mas relocado para um prep step "bake-once" (`group-solids`) que assa `physical_bead_id` no manifest — runtime `extract` não toca mais geometria de interface.
- GUI construída por fork do esqueleto do `sdk-inspection-grouping-optimizer`, não do zero.
- Nada commitado é enviado (push) sem aprovação — engine branch e novo repo GUI ficam locais até revisão.
- No ROADMAP, atribuição de `[4.7]` mantida como estava (não trocar para William, por pedido explícito).
- Shakedown de software (1-2 semanas) encaixado no tempo morto de espera pela prontidão da Hyundai, para não empurrar a data final.
