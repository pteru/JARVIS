## Decisões

- Escopo do jarvis-chat: A+C — Q&A ciente de projeto + memória ambiente;
  sem ações de escrita por ora.
- Política de fala: D — menção estrita (`@JARVIS`), sem modo proativo.
- Continuidade de thread: D — janela sticky ~5min/3 follow-ups, gate por
  formato de pergunta, sem classificador de "assunto encerrado".
- Memória: B — log bruto + fatos destilados (reaproveitando prompt de
  extração do meeting-assistant); 3 categorias adicionais de fatos aceitas.
- Mapeamento projeto↔espaço: arquivo de config explícito.
- Contexto cross-project: A — índice global único, sem barreira; espaços de
  projeto não têm membros externos, então compartilhamento aberto é seguro.
- 100% PT-BR em todo o sistema (detector de pergunta, prompts de resposta e
  extração), consistente com o bot de KB e os relatórios PMO existentes.
- Implementação sem commits intermediários (trabalho direto em `develop`,
  sem worktree) — commit único combinado ficaria para o fim (fases 8.3/9),
  exceto os bugfixes de runtime, que foram commitados à parte.
- Ambos os bots (`kb-chat` e `jarvis-chat`) serão implantados em servidor
  remoto (o mesmo onde roda o `pr-review`), não localmente.
- KB por projeto: PMO é a informação viva/mutável; a KB é regenerada
  periodicamente a partir dele para consistência; formato = markdown em git;
  autoria = pelo próprio JARVIS a partir de conteúdo do PMO + Drive.
