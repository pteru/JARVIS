# 2026-07-13 — PRV10IRIS5: I/O no painel, bloqueios de jog, Redis unificado

## Feito
- **Jog bloqueado — diagnóstico**: caminho de comando OK (audit applied 8-9 ms; JogRefresh ecoado). Bloqueios são do PLC: (1) cadeia de segurança aberta (6 sinais False; Alarm_ActiveWord=63=0b111111, IRIS_FaultCode=6); (2) `Comm_EtherCAT=False` (drives sem barramento; Enable_State1=0 vs 10 nos demais); (3) `ManualMode_Enable=False` e **WSCmd_Mode não existe no PLC** → painel não consegue entrar em manual. Itens p/ automação.
- **Aba I/O no painel** (feito no repo local `workspaces/strokmatic/sdk/iris-bench-panel`, fonte completo Vue3+Vite existe lá): `iomap.py` + `/api/io` (batch Multi-Service via `DirectPLC.read_many`; mock incluso) + `IOView.vue` com 2 visões — **portas físicas** (slot 1 DI16, slots 2-3 AI4-CJ c/ fault por canal, slot 4 DO16-Diag; esquema `Pt00.Data`/`Ch00.Data` confirmado por sondagem) e **por função** (Resistências/Zonas, Barreiras, Segurança, Câmeras, Sensores, Painel — alimentada por ~72 tags nomeadas do programa + canais rotulados). `io_labels.yaml` relido a cada request (automação preenche sem rebuild). Validado: 40/40 canais lendo.
- **Fluxo de deploy sem internet na WS**: base image não resolve → compose monta `backend/` e `frontend/dist` do host; release = build local → rsync → restart. Gotcha: rsync sobrescreveu config.yaml da WS (senha) — restaurado; excluir do rsync futuramente.
- **Poll de I/O**: custo medido/estimado — digitais ~4-8 ms, completo ~15-25 ms por ciclo; proposta de 2 tiers (digitais 150-200 ms, resto 1 s) aguardando OK do Pedro (benchmark interrompido por queda do link local; link voltou).
- **Redis unificado (pedido do Pedro)**: eip-stack migrado do `eip-redis` dedicado p/ o **keydb VK `cache` (host :4000, db0**, senha VK; user `default`). Verificado antes: KeyDB 6.3.4 suporta streams/consumer-groups; zero colisão de namespace (chaves VK: settings/DB_*/`Tags`≠`tags:*`). Compose reescrito (tag-client em host network; serviço redis removido; backups `.bak-eipredis`); painel c/ URL percent-encoded (`VisionKing%40%402022`). `mirror-test.sh`/README apontados p/ container `cache`. **mirror-test PASS 8/8 na instância nova.** Bônus: chaves órfãs da bancada (`*:192.168.15.123`) morreram com o eip-redis.

## Adendo — auditoria 100% do Redis 4000 + realocação p/ db5
- **Censo completo** (CLIENT LIST c/ db= + ss→PID→container + envs de todos os containers): **db0-4 são todos do VK** — db0 tags PLC/speed/captura (`REDIS_DB_PLC=0`, `REDIS_SPEED_DB=0`), db1 cache de imagens (image-saver+camera-acquisition, `Tags` c/ Tag1-3 é deles), db2 inferência, db3 p2o, db4 result/frame-writer. Nossa 1ª escolha (db1) invadia o cache de imagens → **EIP movido p/ db5** (livre): tag-client REDIS_DB=5 (patch bind-mount; repo já suporta REDIS_DB), decoder DST_DB=5, painel /5, mirror-test -n 5. **PASS 8/8 no db5; painel 0/114 null.**
- **hot-decoder NOVO** (`~/eip-stack/hot-decoder/hot_decoder.py`, roda na imagem do tag-client): destrincha `io:in` (128 B, db0) → hash `tags:<key>:hot` (db5) a 100 ms, layout Anexo D/EDS byte a byte; Adapter_Heartbeat = contador de mudanças do blob. Painel HotRedis restaurado (hash flat + updated_at ms) — rota hot viva (staleness ~0,1 s). Candidato a upstream como metade-telemetria da Fase B.
- **Compat downstream**: `part_uuid`/`part_style` (← GM_PVI/GM_Style) adicionados ao tags.json do plc-monitor (db0). **Achado p/ time VK**: image-saver lê `Tags→part_uuid` no **db1** (quem escreve lá é a camera-acquisition, em restart sem câmeras) enquanto plc-monitor publica no db0 — desalinhamento pré-existente do deploy bench; em produção a cam-acq provavelmente propaga. NÃO mexi nos serviços VK.
- Limpeza: chaves EIP órfãs deletadas do db0 (5) e db1 (6); `Tags` VK preservado. Estado final: db0=9 (5 VK + io:in/eds:info/status:comm/schema:version do adapter — C não suporta SELECT, issue upstream), db1=1 (VK Tags), db5=contrato EIP completo.
- io:out:* ainda não existe = **normal**: escritor é a metade-comando da Fase B (encoder); adapter faz GETRANGE a 50 Hz aguardando e envia zeros (seguro por design do EDS).
- Feedback do Pedro registrado em memória: **explicar cada passo antes de executar** em operações multi-comando.

## Adendo 2 — io-decoder configurável (item A do plano de transição)
- **Arquitetura da transição plc-monitor→stack aprovada pelo Pedro**: db0=buffer de imagens (cam-acq escreve, image-saver lê), **db1=stack nova + `Tags` formato VK** (cam-acq lê tags do db1; convenção ORIGINAL VK — `.env` comenta `REDIS_DB_TAGS default: 1`), db4=plc-monitor estacionado até desligar. Decode NÃO é "legacy-mirror": é **feature configurável da stack** (como tags.json do plc-monitor).
- **Investigação prévia**: plc-monitor escreve SÓ `Tags` (MONITOR 10s + código redis_connector); image-saver lê hashes de FRAME (part_uuid carimbado pela cam-acq; REDIS_DB único, default 0); cam-acq (binário C++ genicam_cpp) usa 4 envs: REDIS_DB_CACHE/PLC/MATERIAL/STREAMING (o REDIS_DB=1 do container é resíduo — não existe no binário).
- **io-decoder implantado** (`~/eip-stack/io-decoder/`, substitui hot_decoder): envelope `decoder.json` (source io:in@db0, cold_source p/ merge, N outputs c/ db/key/encode) + **mapas no formato tags.json literal** (endereços `MODULO:O.Data[n].b` resolvem offsets do blob; extensão `cold:grupo/campo` p/ tags fora do assembly — part_uuid←GM_PVI). Saídas ativas: hot canônico db1+db5 (db5 temporário até item B), `Tags`@db1 c/ **encode legacy** (str(True)/cru — 1ª divergência de compat achada e corrigida: plc-monitor grava literal Python, não JSON).
- **Validado**: paridade exata db0×db1 campo a campo; Tag1-3 da cam-acq preservados no hash compartilhado; painel 0/114 null.
- Pendentes (gates do Pedro): B (stack db5→db1 + limpar db5), C (plc-monitor→db4 + script diff), D (envs VK: image-saver REDIS_DB=0; .env REDIS_DB_PLC=1, REDIS_DB_CACHE=0 + recreates), E (validação final + upstream io-decoder no repo strokmatic-eip).
